# 最优架构方案（权威）

> 以技术最优为准。保留原型的页面样式、交互逻辑与核心思想作为需求蓝图，技术实现走本方案。

## 一、系统定性

多游戏品类 + 多级代理分销的博彩盘口平台（仅积分模式）。复杂度集中在三处：

1. **游戏品类异构**：彩票（定时开奖）、棋牌（多人实时对战）、电子（单机即开即结）、街机。
2. **强实时**：定时开奖推送、棋牌多人同步、结果实时回传。
3. **多级分润 + 强一致记账**：每注沿代理链逐级分润，积分绝不错账。

## 二、整体架构

```text
   client-app(APP)      admin-panel(后台)      games-assets(H5游戏)
   Vue3+Capacitor       Vue3+ElementPlus       Phaser3/Canvas
         │                    │                     │(WebView)
         └────────┬───────────┴──────────┬──────────┘
                  │ HTTPS REST            │ WebSocket
         ┌────────▼───────────────────────▼────────┐
         │            server（NestJS · 大脑）        │
         │  HTTP │ WS网关 │ 定时调度 │ 队列(BullMQ)   │
         │  账本·爆率·开奖·多级分润·风控·审核          │
         └────┬──────────────┬──────────────┬───────┘
          PostgreSQL       Redis        对象存储
         (账本/订单)   (实时态/队列/缓存)  (APK/美术)
```

## 三、技术选型

| 层 | 选型 | 理由 |
|----|------|------|
| 工程 | pnpm + Turborepo monorepo | 多端共享类型/契约 |
| 后端 | **NestJS (TS)** | 模块化；内建 DI/WS/Schedule/Queue |
| 实时 | Socket.IO（Nest Gateway） | 棋牌对战、开奖推送 |
| 定时开奖 | @nestjs/schedule + BullMQ | 按期号定时、不漏不重 |
| 异步任务 | BullMQ (Redis) | 结算/分润/推送削峰可重试 |
| 数据库 | PostgreSQL | 账本强一致、事务 |
| 缓存/实时态 | Redis | 在线态/桌局/排行/限流/队列 |
| ORM | Prisma | 类型安全、迁移友好 |
| 客户端 APP | Vue3 + Vite + Pinia + Capacitor | Cursor 内开发，出 APK |
| 游戏 | Phaser3 / Canvas | 独立 H5，WebView 嵌入，只表现不计算 |
| 后台 | Vue3 + Element Plus + ECharts | 表格/图表开箱即用 |
| 语言 | 全栈 TypeScript | 一种语言、类型贯通 |

## 四、后端模块划分

```
auth · users · agents(多级树+分润) · rooms · points(账本) · recharge(上下分)
game-core(引擎抽象) · games/{lottery,table,slot,arcade}
realtime(WS) · scheduler(定时开奖) · activity · cs · admin · risk(风控)
```

## 五、游戏引擎抽象（核心）

统一接口 `IGameEngine`（见 `server/src/modules/game-core/types.ts`），四类各实现一份，结算统一汇入账本+分润：

| 品类 | 驱动 | 结果生成 | 游戏 |
|------|------|----------|------|
| LOTTERY 彩票 | 期号+定时开奖(scheduler) | 服务端按赔率表开奖，整期结算 | ssc/ffc/kuai3/bjsc/lhc/赛车/赛艇 |
| TABLE 棋牌 | 房间→桌→局,状态机,多人 | 服务端发牌/比牌,WS 同步 | 麻将/斗牛/炸金花/龙虎/百家乐 |
| SLOT 电子 | 玩家单次触发,即开即结 | 按 payTable 权重抽倍率 | slots/laba/queen |
| ARCADE 街机 | 下注轮次+倍率轮盘 | 服务端定结果 | bcbm |

新增一类游戏 = 实现 `IGameEngine` + 注册到 `GameEngineRegistry`，不改调用方。

## 六、爆率 / 开奖 / 可证明公平

- 结果 **100% 服务端生成**，前端 `Math.random` 全删，只播动画。
- 爆率/赔率配置版本化（`GameConfig`），后台 `game-control` 可调/灰度/回滚。
- 可证明公平：`serverSeed`(先公布 hash) + `clientSeed` + `nonce`，事后可验证（见 `fairness.util.ts`）。
- RTP 闭环监控 + 风控（异常胜率/大额/RTP 偏离 → `RiskEvent` 告警 + 人工复核）。

## 七、多级代理分润

- `User.parentId` 自引用成树；`User.agentPath` 物化路径（`/p/b/a/`）加速整树查询。
- 每笔有效流水沿代理链向上**逐级分润**（见 `commission.service.ts`），各级按佣金率从平台出账，写 `CommissionRecord`+账本，幂等。
- 代理等级 V1-V5（`AgentLevel`）按团队累计流水晋级，决定佣金率。
- 角色层级：平台(ADMIN) → 分公司(BRANCH) → 代理(AGENT) → 代分(PROXY) → 玩家(PLAYER)，同一棵树用 `role`+`depth` 表达。

## 八、积分账本（唯一记账入口）

唯一入口 + 每笔流水 + `balanceAfter` 对账 + `idempotencyKey` 幂等 + 事务。下注=玩家转平台(BET)，开奖=平台转玩家(WIN)，全程守恒。见 `points.service.ts`。

## 九、实时 + 定时

- 彩票：`scheduler` 按彩种周期投递 BullMQ 任务 → 引擎开奖 → 批量结算 → WS 推送。
- 棋牌：Redis 存桌局状态，WS 广播动作，服务端裁决。
- 断线重连：WS + Redis 状态恢复。

## 十、部署与演进

- 阶段一：单机，Nginx(/api /ws /admin /games /download) + PG + Redis。
- 约束：server 无状态（态进 Redis）、域名配置化 → 将来零改动可拆。
- 演进：先拆「实时游戏网关/开奖引擎」独立扩容；账本/分润永远跟主服务在一起。

## 十一、安全红线

1. 开奖/爆率/积分只在 server；前端随机全删。
2. 可证明公平 + RTP 监控 + 异常风控。
3. 账本走事务+幂等，禁止裸改余额。
4. 数据库不公网；密钥不入库；上线删后门。
5. 多级分润每笔留痕；管理操作进 `AdminLog`。
